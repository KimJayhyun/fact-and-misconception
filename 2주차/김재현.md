## Chapter 3. 타입과 추상화

영국 지하철 노선도의 추상화

- 정확도가 아닌 목적을 위한 추상화

### 추상화를 통한 복잡성 극복

추상화: 현실에서 출발하되 불필요한 부분을 제거하면서 사물의 본질을 드러내는 과정

목적: 불필요한 부분을 제거하면서 현실의 복잡성을 극복

훌륭한 추상화: 목적에 부합해야 함

추상화의 정의

객체지향 프로그래밍은 객체라는 추상화를 통해 현실의 복잡성을 극복

### 객체지향과 추상화

#### 그룹으로 나누어 단순화하기

일반적인 외형과 행동 방식을 가지고 있고 차이점을 무시하면 그룹화할 수 있음

- 복잡성을 효과적으로 감소

#### 개념

공통적인 특성을 기준으로 객체를 여러 그룹으로 묶어 동식에 다뤄야하는 가짓수를 줄임으로써 상황을 단순화함

개념(Concept): 공통점을 기반으로 객체들을 묶기 위한 그릇

개념을 이용하면 객체를 여러 그룹으로 뷴류(Classification)할 수 있다.

개념의 인스턴스

객체란 특정 개념을 적용할 수 있는 구체적인 사물
개념이 적용되었을 때, 객체는 개념의 인스턴스라고 한다.

#### 개념의 세 가지 관점

1. 심볼(symbol): 개념을 가리키는 간략한 이름이나 명칭
2. 내연(intension): 개념의 완전한 정의, 내연의 의미를 통해 객체가 개념에 속하는지 알 수 있음
3. 외연(extension): 개념에 속하는 모든 객체의 집합

개념과 추상화, 객체지향의 관계성

### 타입

#### 타입은 개념이다

개념 = Type

#### 데이터 타입

1. 타입은 데이터가 어떻게 사용되냐를 결정하는 것
2. 타입에 속한 데이터를 메모리에 어떻게 표현하는지는 외부로부터 철저하게 감춰짐

데이터 타입의 정의

#### 객체와 타입

객체지향 프로그래밍 작성 시, 객체를 일종의 데이터처럼 사용

-> 객체를 타입에 따라 분류하고 타입에 이름을 붙힘

-> 프로그램에서 사용할 새로운 데이터 타입을 선언하는 것 = 객체지향 프로그래밍

객체는 데이터인가?? No

객체에서 중요한 것은 상태가 아닌 행동

객체의 타입에 대한 2 가지 사실

1. 객체가 어떤 타입에 속하는지 결정하는 것은 객체가 수행하는 행동

   어떤 객체들이 동일한 행동을 한다면 같은 타입으로 분류할 수 있음

2. 객체의 내부적인 표현은 철저희 외부로부터 감춤

   객체의 행동을 효과적으로 수행할 수 있다면 객체 내부 상태를 어떤 방식으로 표현해도 무방

#### 행동이 우선이다.

내부적인 표현 방식이 다르더라도 객체들이 동일한 행동을 한다면 동일한 타입이다.

-> 같은 행동을 하고, 내부적인 표현은 외부로부터 감추기 때문에 중요하지 않음

-> 객체의 타입을 결정하는 것은 객체의 행동뿐임

동일한 타입인 객체 -> 동일한 행동 -> 동일한 책임 -> 동일한 메세지 수신 -> 메세지를 처리하는 방식은 다를 수 있음

-> 다형성: 동일한 요청에 다른 방식으로 응답할 수 있는 능력

-> 캡슐화

데이터를 결정하고 객체의 책임을 결정하면 유연하지 못 하다 = 데이터 주도 설계(Data Driven Design)

-> 책임-주도 설계(Responsibility Driven Design)

### 타입의 계층

일반화 / 특수화 관계

#### 일반화 / 특수화 관계

일반화 / 특수화 관계를 결정하는 것 또한 객체의 행동

일반적인 타입이 외연의 크가기 더 크지만, 행동 수가 적다.

특수한 타입이 외연의 크기가 더 작지만, 행동 수가 많다.

#### 슈퍼타입과 서브타입

행동에 따라 결정됨.

어떤 타입을 다른 타입의 서브타입이라고 한다면 다른 타입을 대체할 수 있어야 한다.

-> 서브타입이 행동 수가 더 많으니까

#### 일반화는 추상화를 위한 도구이다.

일반화 / 특수화 게층은 객체지향 프로그래밍에서 추상화의 두 번째 차원을 적절하게 활용한 대표적인 예시

객체지향 프로그래밍은 거의 대부분 세상을 분류와 일반화 / 특수화 기법을 동시에 적용한다.

### 정적 모델

#### 타입의 목적

시간에 따라 동적으로 변하는 객체가 가진 복잡성을 타입으로 극복

#### 그래서 결국 타입은 추상화다.

타입이라는 추상화를 통해 시간과 상태 변화를 제거하고 정적인 관점에서 객체를 볼 수 있다.

객체의 동적인 특성을 추상화할 수 있다.

동적인 특성 = 객체의 시간의 따른 상태의 변화

객체의 복잡성을 단순화할 수 있음

#### 동적 모델과 정적 모델

1. 객체가 특정 시접에 어떤 상태를 가지느냐: 객체의 스냅샷(Snapshot)

   객체지향 모델링을 위한 표준언어인 UML에서 스냡샷은 객체 다이어그램(object diagram)이라고도 한다.

   또한 시간에 따른 상태의 변화와 어떤 행동을 하는지를 포착하는 것이 동적 모델(dynamic model)이라고 한다.

2. 객체가 가질 수 있는 모든 상태와 모든 행동을 독립적으로 표현하는 것을 타입 모델(type model)이라고 한다.

   객체가 속한 타입의 정적인 모습을 표현하기 때문에 정적 모델(static model)이라고도 한다.

객체지향 프로그래밍을 하는 중에는 정적 모델, 런타임에 디버깅을하는 경우에는 동적 모델을 탐험한다.

#### 클래스

정적인 모델은 클래스로 구현이 된다.

-> 타입을 구현하는 보편적인 방법이 클래스를 이용하는 것이다.
